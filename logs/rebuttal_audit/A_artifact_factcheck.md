## Discrepancies & missing artifacts

### D1 — Claim #9 (tuning): recovery shape is mis-attributed (PARTIAL, inline-author-note focus)
The §2 mapping table (L188) says: *"§7.3 adds GROUP_SIZE_M L2-swizzle / num_warps / num_stages **at 4096²** (recovers **~32%→98%**)."*
`autotune_matmul.csv` shows the **32.23%→98.49% recovery is at 16384²**, not 4096²:
- **16384²:** triton_plain e_vs_cublas=**32.23**, triton_autotune=**98.49** (speedup 3.056×).
- **4096²:** triton_plain=**82.98**, triton_autotune=**102.21** (speedup 1.232×).

`exp_autotune_matmul.py:165` itself labels 16384² the **"Δ~=0pp regime (§5)"** and 4096² the "§7.3 1.66× regime". So the rebuttal pins the dramatic 32→98 recovery to the shape its own script calls the §5 Δ≈0 regime. The intended story (script docstring L22-26) is: at 4096² the expanded search wins modestly; at 16384² the plain heuristic config is *already near-optimal so expanded search adds ~0* — which is the opposite of "32%→98% recovery at 4096²". The grid definition (`configs.py:19-23`, 12 configs, bm/bn∈{32,64,128}, bk∈{32,64}) and the GROUP_SIZE_M/num_warps/num_stages list (`matmul_triton.py:10-23`) are both correct; only the shape↔recovery pairing in the rebuttal prose is wrong.

### D2 — Claim #9b: "1.66×" is unreproduced by the artifact (PARTIAL)
The 1.66× §7.3 speedup is the **paper's** assertion the experiment was built to reconcile (`exp_autotune_matmul.py:11,23`), not a measured result. Measured autotune-vs-plain = **1.232× at 4096²** and **3.056× at 16384²**. The rebuttal text avoids stating 1.66× numerically (and §2-§3 say "all numbers live in the artifact"), so this is not a direct overclaim in the prose, but a reviewer recomputing from `autotune_matmul.csv` will not find 1.66× at 4096². Logs (`RIGOR_AUDIT.md:35`, `CONSISTENCY_AUDIT.md:150,155`) acknowledge the §5-vs-§7.3 1.66× tension.

### D3 — Claim #10: the paper's "94.6% vs 97.8%" pair does NOT map to softmax 95.2% (PARTIAL)
- REBUTTAL §3 (L227-228): the paper's pair "maps to **layer_norm (94.5%)** and **softmax (95.2%)**."
- Measured `significance.csv`: layer_norm triton **94.46** (≈94.5 ✓), softmax triton **95.18** (≈95.2).
- **But the paper's second number is 97.8%, and softmax measures 95.2%** — a 2.6pp gap. The layer_norm side (94.6→94.46) is clean; the softmax side (97.8→95.2) is not a clean match.
- Note an **internal inconsistency**: `exp_significance.py:9` maps the paper pair to *"softmax (97.6%)"*, whereas the rebuttal (correctly per measurement) uses 95.2%. The script comment (97.6%) and the rebuttal (95.2%) disagree on which historical softmax number the paper's 97.8% refers to. Either way, the locked-run softmax (95.2%) does not equal the paper's 97.8%, so the assertion "that pair of numbers maps to layer_norm/softmax" is only half-supported. The conclusion that both "resolve as genuinely below parity" still holds (verdicts in CSV: "significant: DSL slower" for both).

### D4 — Claim #2 (RC0b code-gen) is an attribution, not a counter (PARTIAL)
The LDG.128/vectorization deficit is measured (conv load-eff 36.35% vs cuBLAS 99.66%), but "code-generation issue" is an interpretive label; no counter in the artifact distinguishes "code-gen origin" from other causes. Low-risk (consistent with data) but not directly measured.

### D5 — Claim #15 (FlagGems provenance) lacks any explicit attribution (PARTIAL)
layer_norm=TorchInductor is provable from imports. "argmax/max_reduction/log_softmax = FlagGems idioms" rests on idiom-style function names (e.g. `can_use_int32_index`) with **no FlagGems attribution comment** in any source file. Idiom-consistent but not hard-provenance; the rebuttal itself softens this to "follow FlagGems idioms" / "partly recoverable," so prose is defensible, artifact proof is weak.

### Minor consistency note (not a contradiction)
- Small-shape conv mitigation (`conv_mitigation_small.csv`: 3×3=45.74%, 1×1=34.99%) is *lower* than the "57–69%" headline; the rebuttal correctly scopes the 57–69% to the large/realistic shape, so no contradiction — just be sure the paper states the shape.
